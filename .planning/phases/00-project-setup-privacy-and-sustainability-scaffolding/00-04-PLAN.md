## Load-bearing architectural rules

These two rules are non-negotiable and code-review enforced. Violating either is a release blocker.

### Rule 1: Helper never asserts game state

The helper CLI emits raw observed signals only. It does not compute XP, evaluate quests,
unlock achievements, or assert level progress. The backend is the sole authority on game state.

- The helper writes events to a local SQLite outbox.
- The flusher daemon batches them to `/v1/events` on the backend.
- Backend computes XP, evaluates quest rules, fires achievement triggers.
- Backend pushes computed state down to the helper for display (XP totals, active quests, etc.).
- The helper renders state from the backend; it never invents state.

This rule is enforced by code review on every PR touching `apps/cli` (Phase 5+ owns implementation).
The rationale: a helper that asserts game state is trivially gameable -- a user could spoof XP,
quest completions, or achievement unlocks. Centralized authority on the server (with anti-cheat
hardening in Phase 10) makes the system honest.

### Rule 2: Anthropic OAuth token never reaches the backend

The OAuth token at `~/.claude/.credentials.json` (used by the helper to call
`https://api.anthropic.com/api/oauth/usage` for 5h/7d window data) STAYS on the user's machine.

- The helper extracts metadata from the token (model, token-counts, session boundaries) and
  sends ONLY the metadata to the backend.
- The token bytes themselves never enter `/v1/events`, never enter our backend logs, never
  enter our backend Postgres. Period.
- Phase 1 installs a Pino redaction transport with filters for the `sk-ant-*` token prefix
  and any bearer-token-shaped header (PRIV-05 policy stated here, installation deferred).
- Phase 1 also installs a CI grep gate that fails any commit containing `sk-ant-` substring
  (INGEST-10 was originally tagged Phase 3 but is installed in Phase 1 alongside the logger).
- Phase 5 enforces this in the helper code (HELPER-12).

The rationale: Anthropic OAuth tokens are bearer credentials with broad scope. Aggregating
them in any backend creates a high-value target for attackers and a regulatory liability for us.
Pushing the rule to "token never leaves the local machine" eliminates the entire class of
backend-side credential leak.

## Cross-cutting policy: redaction-first logging

Pino is the standard backend logger (Phase 1 installs). Its redaction transport is configured
BEFORE any handler is registered. Redacted fields:

- Any string matching `sk-ant-[A-Za-z0-9_-]+` (Anthropic OAuth tokens)
- Any string in headers named `Authorization`, `Cookie`, `Set-Cookie`, `X-Api-Key`
- Any field named `password`, `password_hash`, `email`, `ip`, `device_token`

This applies to both successful request logs and error logs. PII (email, IP) is hashed where
needed for audit; raw values never enter logs.

## Cross-cutting policy: Plausible Cloud only

The public hosted instance uses **Plausible Cloud** (managed) for analytics. The Plausible
Community Edition is AGPL-3.0 and would attach to anyone running our distributed Docker
bundle (Pitfall 2 in `00-RESEARCH.md`). Therefore:

- `docker-compose.yml` MUST NOT contain a Plausible CE service.
- The web app embeds Plausible's `script.js` (MIT-carved-out) on rendered pages -- no AGPL
  contamination.
- Self-hosters who want analytics can run their own Plausible instance separately (separate
  compose file, separate domain) at their option. Our distribution stays Apache-2.0-clean.
